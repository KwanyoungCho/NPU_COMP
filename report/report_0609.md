# TVM 기반 NPU 컴파일러 — 구현 계획 및 진행상황

> 작성일: 2026-06-09
> 대상 모델: **Llama 3.2 3B** / 대상 하드웨어: 본 레포의 NPU c-model(`_poc/mysim`)
> 본 문서는 **코드 없이도** 처음 보는 사람이 이해하도록 동작 위주로 서술한다.

---

## 0. 개요

목표는 **신경망(Llama 레이어)을 NPU가 실행할 명령어로 자동 변환하는 컴파일러**를 만드는 것이다.
컴파일러 프레임워크로 **TVM(Relax)** 을 쓰고, 결과는 **주어진 시뮬레이터 mysim**에서 실행·검증한다.

한 문장 요약:
> 모델을 TVM Relax 그래프로 받아 → NPU가 지원하지 않는 연산을 지원 연산 조합으로 바꾸고 → 64×64 타일로 쪼개 → NPU 명령어로 생성 → mysim에서 돌려 정답과 대조한다.

현재까지 **단일 Llama 레이어를 끝까지 컴파일·실행해 부동소수점 참조와 상대오차 0.13%로 일치**시켰고, **임의 차원을 64×64로 타일링**하는 것까지 동작한다(아래 §4~5).

## 1. 전체 구조: 커스텀 가속기를 컴파일러에 붙이는 방법

이 장은 "신경망 → 가속기 명령어"를 만드는 **일반적인 방법들**과, 그중 우리가 어디에 서 있고 **어디로 가야 하는지**를 설명한다.

### 1.1 컴파일의 두 단계 (IR 레벨)

신경망은 두 가지 높이로 표현된다.

- **Relax (그래프 레벨)** — "무엇을 하는가". 예: `y = matmul(x, w)` 한 줄. 연산들의 그래프.
- **TIR (루프 레벨)** — "어떻게 계산하는가". 위 한 줄을 실제 3중 for문으로 펼친 것:
  "출력의 각 칸마다, 숫자 하나씩 곱해서 더한다"는 반복문.

표준 TVM은 보통 `Relax → TIR → 기계어(CPU/GPU)` 순으로 내려간다. 우리 NPU는 이 흐름을 그대로 따를 수도, 우회할 수도 있다. 그 선택지가 아래 둘이다.

### 1.2 방법 A — BYOC (Bring Your Own Codegen)

**아이디어**: 모델 그래프에서 **내 가속기가 지원하는 부분만 떼어** 내 코드 생성기로 보내고, **나머지는 CPU**가 처리하게 한다.

- 동작: ① 내 가속기가 지원하는 연산 **패턴을 등록** → ② TVM이 그래프에서 그 패턴을 찾아 **묶음으로 분리(partition)** → ③ 분리된 부분을 **내 코드 생성기**로, 나머지는 CPU로.
- 비유: 일반 주방(CPU)에서 요리하다가 "면 뽑기는 저 전용 기계(가속기)에 맡겨" 하고 그 작업만 떼주는 것.
- **언제 빛나나**: 실제 모델(ONNX/HuggingFace 등)은 가속기가 못 하는 연산(임베딩 룩업, 형태 변환 등)이 섞여 있다. BYOC는 그걸 **CPU로 자동 우회**해 준다. 그래서 **"표준 모델을 받아 가속기에 올리는" 정석 경로**다(TensorRT·CUTLASS 등이 이 방식으로 TVM에 붙는다).

### 1.3 방법 B — TIR + tensorize

**아이디어**: 연산을 TIR(루프)로 내린 뒤, 안쪽 64×64 블록을 **가속기의 행렬 명령 하나로 치환**한다.

- 동작: ① 64×64 행렬곱의 "의미"와 "하드웨어 명령"을 한 쌍으로 정의(intrinsic) → ② 큰 행렬곱을 64×64로 타일링 → ③ 안쪽 블록을 그 명령으로 **교체(tensorize)**.
- 비유: 반죽을 손으로 1000번 젓는 루프 대신 "이 부분은 반죽기(명령 1개)로" 바꿔치기.
- **언제 빛나나**: TVM이 **타일링·스케줄링을 자동으로** 해주고 자동 튜닝까지 된다. GPU 텐서코어, TVM 공식 가속기 예제(VTA)가 이 방식을 쓴다.

### 1.4 우리의 현재 방식 — operator-level "lite"

우리는 위 둘의 프레임워크를 **쓰지 않고**, Relax 그래프를 직접 순회하며 각 연산을 NPU 명령으로 바로 바꾼다.

- 이유: **이 NPU는 coarse-grained**(행렬곱·벡터연산이 통째로 명령어 1개)라, TIR 스칼라 루프로 잘게 풀었다가 다시 합치는 건 빙 돌아가는 길이다. 그래프 연산 = 명령어로 직행이 자연스럽다.
- 성격: 사실상 **"프레임워크 없는 BYOC"**. 핵심 아이디어(그래프 연산을 내 코드로 매핑)는 같지만, 패턴 등록·분리·런타임 통합 같은 격식을 다 뺀 최소 버전이다.
- 장점: 빠르게 동작·검증(현재 byte-exact까지 확인). 단점: TVM의 자동 스케줄링·CPU 우회·모델 자동 수입을 못 받는다.

### 1.5 결론 — 우리가 가야 할 방향

원칙: **모델에 가까울수록 TVM에 기대고, 하드웨어에 가까울수록 우리가 만든다.**

| 계층 | TVM이 해주나 | 방향 |
|---|---|---|
| 모델 수입 (ONNX/HF/PyTorch → Relax) | ✅ 완성됨 | **TVM 프런트엔드/MLC 적극 활용** |
| 그래프 최적화·메모리 플래닝 | ✅ 있음 | 점진적으로 활용 |
| 미지원 연산 분해(legalize) | ◐ 일부 | TVM + NPU 고유 우회 |
| 타일링/스케줄(tensorize) | ◐ 조건부 | 정식 백엔드화 시 고려 |
| **NPU 명령어 생성 + 실행** | ❌ 불가(커스텀 HW) | **반드시 우리 것** |

- **BYOC로 가는 게 맞다.** 실제 ONNX/HF 모델을 받는 순간, BYOC의 "지원 부분만 떼고 나머지는 CPU" 구조가 정확히 필요하다. 그리고 지금 우리 코드 생성기는 그때 **BYOC의 등록 백엔드로 그대로 재사용**되므로 버려지지 않는다. 즉 현재의 lite는 BYOC의 전 단계다.
- **tensorize는 조건부다.** coarse 가속기엔 잘 맞는 기술이지만, 우리 타깃(mysim)은 ⓐ 루프 명령이 없어 어차피 펼쳐야 하고, ⓑ TVM엔 우리 NPU용 TIR→명령어 생성기가 없어 결국 더 어려운 코드 생성을 새로 짜야 한다. 따라서 **이 프로젝트를 정식 TVM 백엔드(자체 런타임·자동튜닝)로 키울 때** 도입하는 것이 맞다.
- 요약: **프런트(모델 수입)는 TVM/BYOC를 쓰고, 백엔드(NPU 명령 생성)는 우리가 유지**하는 그림으로 수렴한다. 지금은 백엔드를 검증하는 단계이므로 lite가 합리적이고, 실제 모델 단계에서 BYOC로 확장한다.

## 2. 타깃 NPU와 제약 (컴파일러가 다뤄야 하는 현실)

- **mysim이 유일한 실행기이자 기준**이다. NPU의 동작을 정의하는 c-model이며, 우리는 이걸 타깃으로 명령어를 만들 뿐 실행기를 새로 만들지 않는다. 또한 **현재 mysim은 수정 불가(고정)**로 둔다.
- **숫자 정밀도(FP16)**: 내부 연산은 정확히(float32) 하지만 **메모리에 저장할 때만 FP16로 반올림**한다. → "언제 저장하느냐"에 따라 결과가 미세하게 달라진다(§4.3의 타일링에서 핵심이 됨).
- **메모리(G-buffer)**: 모든 데이터가 사는 1차원 배열. **동적 할당이 없어** 모든 텐서 주소를 컴파일 타임에 미리 정해야 한다.
- **명령어 제약**: 행렬 연산기는 **64×64**가 한 번에 처리하는 최대 크기. **루프/분기 명령이 없어** 반복은 전부 펼쳐야(unroll) 한다. 전치·합·브로드캐스트·표준 활성화(SiLU)·음수화 등 **여러 연산이 ISA에 없다**.
- **현재 스코프(의도된 단순화)**: 우선 **prefill**만 동작시키고, KV 캐시는 저장만, softmax의 최댓값 빼기는 생략(작은 점수에서 안전), 전치는 원소 복사로, 누적은 펼침으로 처리한다. (각 항목은 향후 ISA 확장 시 재검토.)

## 3. 우리가 구현한 lowering — 무엇이 필요했고 어떻게 했나

전체 파이프라인은 다음과 같다. 각 단계가 곧 하나의 구성요소다.

```
 모델(Relax 그래프)
   │  ① 메모리 배치   : 텐서마다 G-buffer 주소 정하기
   │  ② legalize     : NPU에 없는 연산을 있는 연산 조합으로 바꾸기
   │  ③ 코드 생성     : 각 연산을 NPU 명령어로 (필요하면 64×64 타일링)
   ▼
 NPU 명령어 + 초기 데이터
   │  ④ 런타임        : 주어진 mysim 실행
   ▼
 결과  ──(⑤ 검증)── 부동소수점 참조와 대조
```

구성요소별 역할(코드 아님, 하는 일):

| 구성요소 | 하는 일 |
|---|---|
| **명령어 인코더** | 사람이 읽는 명령("주소 5에서 8개 로드")을 mysim이 이해하는 32비트 숫자로 번역. *실행기가 아니라 번역기.* 실제 NPU 예제 바이너리 **14,336개와 1비트도 안 틀리게** 일치함을 확인. |
| **런타임** | 명령어·데이터 파일을 만들고 **주어진 mysim을 실행**한 뒤 결과를 읽음. 동작은 우리가 흉내내지 않음. |
| **메모리 배치** | 입력·가중치·상수·중간결과에 G-buffer 주소를 앞에서부터 차곡차곡 배정. 상수(ones 등)는 실행 전 미리 채워 둠. |
| **코드 생성** | Relax 연산을 NPU 명령으로 매핑. 행렬곱·원소연산은 그대로, 큰 행렬곱은 64×64 타일로 쪼갬. |
| **legalize(우회)** | ISA에 없는 연산을 있는 연산 조합으로 분해(아래 표). |
| **타일링** | 64×64를 넘는 차원을 조각내고 부분곱을 누적. |

**미지원 연산을 어떻게 우회했나** (이게 컴파일러의 알맹이):

| 필요한 것 | NPU엔 없음 → 우회 방법 |
|---|---|
| 합(reduce) | **1로 채운 벡터와의 행렬곱** (내적이 곧 합) |
| 복제(broadcast) | **1 벡터와의 행렬곱** (외적으로 값 복제) |
| 전치(transpose) | **원소 단위 복사** (주소만 바꿔 옮김) |
| SiLU 활성화 | `z/(1+exp(-z))`를 exp·덧셈·나눗셈·곱셈으로 조합 |
| 음수화 | **0에서 빼기** (0 − x) |
| RoPE 회전 | cos/sin 표는 **미리 계산해 적재**, 절반 회전은 **고정 순열행렬과의 곱**으로 |
| Softmax | exp → 합 → 복제 → 나눗셈 (최댓값 빼기는 생략) |

→ 이렇게 **행렬곱·원소연산·제곱근·exp** 라는 소수의 기본 연산만으로 RMSNorm·Attention·SwiGLU 전체를 구성했다.

아래에서 핵심 네 단계(메모리 배치 · legalize · 코드 생성 · 타일링)가 **구체적으로 어떻게 동작하는지** 설명한다.

### 3.1 메모리 배치 (Memory planning)

**왜 필요한가**: NPU엔 동적 할당이 없다. 모든 텐서(입력·가중치·상수·중간결과)는 G-buffer라는 **1차원 배열의 고정된 주소**에 미리 놓여 있어야 한다.

**어떻게 동작하나**: 가장 단순한 **앞에서부터 차곡차곡(bump) 방식**이다. 그래프를 훑으며 텐서마다 그 크기(원소 개수)만큼 자리를 떼어 시작 주소를 부여한다.
- 예: `x[2,3]`은 주소 0부터 6칸, `w[3,2]`는 주소 6부터 6칸, 결과 `[2,2]`는 주소 12부터 4칸.
- 2차원 행렬은 **행 우선(row-major)**으로 펼쳐 담는다: `[i,j]` → 주소 `시작 + i×(열 수) + j`. 이 규칙 덕분에 나중에 "몇 번째 행의 몇 번째 열"을 주소로 바로 계산할 수 있다(타일링의 주소 계산이 여기 기반).
- **별칭 처리**: "결과를 그대로 출력으로 내보내는" 경우처럼 *새 계산이 아니라 다른 이름일 뿐*이면, 새 자리를 주지 않고 **같은 주소를 공유**한다.
- **상수**(ones·마스크·RoPE 표 등)는 그 데이터를 기억해 두었다가, 실행 직전 초기 G-buffer에 미리 써 넣는다.

**핵심**: 이 "주소표"가 이후 모든 단계의 *어디서 읽고 어디에 쓸지*를 정한다. 텐서 간 연결이 전부 주소로 이뤄지므로, 메모리 배치가 컴파일의 토대다.

### 3.2 legalize (미지원 연산 우회)

**왜 필요한가**: 신경망엔 NPU 명령에 없는 연산(합·복제·RMSNorm·Softmax·SiLU·RoPE)이 있다. 이를 **NPU가 가진 기본 연산들의 조합 "레시피"로 바꾸는** 단계다.

**어떻게 동작하나**: 핵심은 "없는 연산을 행렬곱·원소연산으로 환원"하는 두 트릭이다.
- **합 = ones와의 행렬곱**: `[행, K]` 행렬을 `ones[K,1]`과 곱하면, 각 행의 모든 값을 더한 `[행, 1]`이 나온다(내적이 곧 합).
- **복제(broadcast) = ones와의 행렬곱**: `[행, 1]`을 `ones[1, N]`과 곱하면 값이 N칸으로 복제된 `[행, N]`이 된다(외적).

이 두 트릭으로 고수준 연산을 풀어낸다. 예를 들어 **RMSNorm 한 줄**은:
> 제곱 → (ones곱으로) 행 합 → 평균(상수 곱) → 제곱근 → 1/rms(나눗셈) → (ones곱으로) 복제 → x와 곱 → 가중치와 곱

즉 **기본 연산 약 10개**로 펼쳐진다(§4.2에서 실제 출력으로 확인). Softmax·SiLU·RoPE도 같은 식으로 exp·나눗셈·곱·고정 상수 행렬곱으로 환원된다.

**현재 상태(정직)**: 지금은 이 레시피를 *자동 변환 패스*가 아니라 **그래프를 만들 때 직접 호출하는 빌더**로 구현했다. 실제 모델을 수입할 때는 "수입된 연산을 자동으로 이 레시피로 바꾸는 패스"로 교체해야 한다(보고서 §6의 다음 단계).

**핵심**: legalize 덕분에 코드 생성기는 **소수의 기본 연산만** 알면 된다.

### 3.3 코드 생성 (Codegen)

**왜 필요한가**: 기본 연산 그래프를 **실제 NPU 명령어 시퀀스**로 바꾸는 단계다.

**어떻게 동작하나**: 그래프를 순서대로 훑으며 각 연산을 **정해진 "명령 묶음"으로** 찍는다. NPU는 "상태를 세팅하고 그다음 실행"하는 머신이라 패턴이 일정하다.
- **행렬곱** → `크기 설정(TILE) → 주소 설정 → A 로드 → B 로드 → 행렬곱 → 주소 설정 → 저장`
- **원소 연산** → `길이 설정(VLEN) → 로드 → (로드) → 연산 → 저장`
- 이때 쓰는 주소는 §3.1의 주소표에서 가져온다. **주소 설정 명령 바로 다음 로드/저장이 그 주소를 사용**한다.
- 마지막에 종료(HALT) 명령을 붙인다.

예: `matmul(x, w)` 하나가 §4.1의 13개 명령으로 펼쳐진다.

**핵심**: "연산을 알아보고 → 정해진 명령 템플릿을 찍는" 방식이다. 이게 가능한 건 NPU가 **coarse-grained**(연산 하나 = 명령 하나)이기 때문이다. 그래서 연산을 잘게 풀 필요 없이 그래프 레벨에서 바로 명령으로 간다.

### 3.4 타일링 (코드 생성의 확장)

**왜 필요한가**: 실제 행렬은 PE 크기(**64×64**)보다 훨씬 크다. 한 번에 처리할 수 없으니 **조각내서** 해야 한다.

**어떻게 동작하나**: 출력을 64×64 타일로 나누고, 각 타일을 K(공통) 방향으로 누적한다.
- 출력 `C[M,N]`을 64×64 조각들로 분할한다.
- 각 출력 조각마다: K를 64씩 끊어 **부분곱**을 구하고 **더해 나간다(누적)**.
- 누적할 명령이 ISA에 없어, **"저장 → 다시 로드 → 더하기"** 로 누적한다. 이때 **저장할 때마다 FP16 반올림**이 일어나, 한 번에 한 결과와 미세하게 달라진다(§4.3에서 실제로 확인 — 버그가 아님).
- **데이터 모으기(gather)**: 행렬을 조각낼 때, 어떤 조각은 메모리에서 **띄엄띄엄(strided)** 떨어져 있다(예: A의 K 조각은 각 행의 일부라 행 간격만큼 떨어짐). NPU 로드는 연속만 읽으므로, 그런 조각은 **행마다 연속 버퍼로 복사해 모은다**. 반대로 이미 **연속이면 복사를 생략**한다.
- 예: `K=128`이면 64씩 2조각으로 나눠 부분곱 2개를 누적.

**현재 상태**: M·N·K 어떤 차원이든 동작하며, **모든 행렬 명령이 64×64 이하**(hardware-legal)임을 자동 확인한다. "연속이면 복사 생략" 최적화를 적용해, 작은/연속 구간은 효율적이다. 남는 복사 비용은 **NPU에 strided-load 명령이 없어서** 생기는 본질적 비용이다(N이 64를 넘어 열을 쪼갤 때 등). **진행 중**: 실제 3B 차원의 비용 분석, 추가 최적화(스크래치 버퍼 재사용 등).

**핵심**: 큰 연산을 하드웨어가 한 입에 처리할 크기로 쪼개 누적한다. FP16 누산 때문에 검증은 *같은 반올림 순서를 흉내낸 참조*와 대조해야 한다.

## 4. walkthrough 예제로 보는 실제 동작

코드 대신 **실행 결과**로 각 단계가 무엇을 하는지 본다. 세 가지 학습 스크립트가 있고, 아래는 그 출력을 정리한 것이다.

### 4.1 기본 흐름 — 작은 행렬곱 `C[2,2] = A[2,3] @ B[3,2]`

`walkthrough_matmul.py`를 돌리면 6단계가 차례로 보인다.

- **① 그래프**: `lv = matmul(x, w)` 한 줄. (아직 "무엇"만 있고 명령어 아님)
- **② 메모리 배치**: `x→주소 0`, `w→주소 6`, `결과→주소 12`. (출력 변수 `gv`는 `lv`의 별칭이라 같은 주소 12를 공유)
- **③ 코드 생성**: 그 한 줄이 **TILE → 주소설정 → LOAD → MATMUL → SAVE** 명령들로 펼쳐진다. NPU는 "먼저 크기·주소를 세팅하고 그다음 로드/연산/저장"하는 머신이다.
- **④ 데이터 채우기**: 주소 0에 A=[1,2,3,4,5,6], 주소 6에 B를 써넣음.
- **⑤ 실행**: mysim이 돌고 주소 12~15에 `[4, 5, 10, 11]`이 채워짐.
- **⑥ 검증**: numpy 정답 `[[4,5],[10,11]]`과 **완전히 일치(byte-exact)**.

핵심 관찰: 모든 텐서 간 연결이 **G-buffer 주소**로 이뤄진다. 그래서 메모리 배치가 중요하다.

### 4.2 legalize — RMSNorm이 어떻게 분해되나

`walkthrough_rmsnorm.py`는 `rms_norm()` **한 번 호출이 기본 연산 10개로 펼쳐지는 것**을 보여준다(요약):

```
lv  = x * x                              (제곱)
lv1 = matmul(lv, ones[D,1])              ← 합(reduce): ones와 곱해 행마다 더함
lv2 = lv1 * (1/D)                        (평균)
lv3 = sqrt(lv2)
lv4 = ones / lv3                         (1/rms)
lv5 = matmul(lv4, ones[1,D])             ← 복제(broadcast): ones와 곱해 펼침
lv6 = x * lv5                            (x/rms)
lv7 = matmul(ones[seq,1], w)             ← 복제(broadcast): 가중치 펼침
lv8 = lv6 * lv7                          (· 가중치)
```

- 전용 명령이 없는 **합과 복제가 전부 "ones와의 행렬곱"으로 바뀐 것**이 핵심이다.
- 검증: NPU 결과 `[0.1631, -0.2184, 0.9819, …]` vs 정답 `[0.1631, -0.2184, 0.9823, …]`, **최대오차 0.00115** (FP16 반올림 수준).

### 4.3 타일링 — 64×64로 쪼개기와 FP16 누산

`walkthrough_tiling.py`는 같은 행렬곱(K=128>64)을 두 방식으로 비교한다.

- **(A) 한 방에(logical)**: MATMUL 명령 **1개**(총 13명령). mysim은 K=128도 받아주지만, 실제 64×64 PE에선 불법.
- **(B) 타일링(hardware-legal)**: K를 64씩 쪼개 부분곱을 누적. **모든 행렬 명령이 64 이하**임을 자동 확인.

결과 비교(중요):
- 타일링 결과는 **같은 반올림을 흉내낸 참조(`tiled_fp16_ref`)와 byte-exact로 일치**.
- 하지만 **한 방에 한 결과와는 16칸 중 8칸이 다르다.** 이는 **버그가 아니다** — 조각마다 저장할 때 FP16 반올림이 끼어들기 때문이며, 둘 다 진짜 정답(고정밀)엔 가깝다(상대오차 4.1e-4).

이 예가 §2의 "저장 시 반올림 → 언제 저장하느냐로 결과가 달라진다"를 실제 숫자로 보여준다. 그래서 타일링 결과는 한 방에 한 결과와 비교하면 안 되고, **같은 반올림 순서를 모델링한 참조**와 비교해야 한다.

## 5. 검증 결과

모든 결과는 **독립 구현한 부동소수점 참조와 교차 비교**했다.

| 항목 | 결과 |
|---|---|
| 명령어 인코더 (실제 바이너리 14,336워드) | **byte-exact** 일치 |
| 행렬곱 / 원소연산 | **byte-exact** |
| RMSNorm / SwiGLU / Attention(단일헤드) | 상대오차 ~0.001 (FP16 수준) |
| **전체 Llama 레이어**(RMSNorm→GQA+RoPE+causal→잔차→RMSNorm→SwiGLU→잔차) | **상대오차 0.13%** |
| 타일링(임의 차원, M·N·K>64 포함) | 같은 반올림 참조와 **byte-exact**, 모든 타일 ≤64×64 |

추가 정량 분석:
- **전치 오버헤드**: 단일헤드 attention에서 원소복사 전치가 **전체 명령의 68%**를 차지. → 전치/strided 로드 명령이 ISA에 추가되면 큰 이득이라는 정량 근거.
- (참고) 타일링은 **연속이면 복사를 생략**하고 **진짜 띄엄띄엄(strided)일 때만 gather/scatter**한다(예: N>64라 열을 쪼갤 때). 즉 작은 차원/연속 구간은 효율적이고, 남는 복사 비용은 NPU에 **strided-load 명령이 없어서 생기는 본질적 비용**이다(전치 오버헤드와 같은 맥락 → 해당 ISA 추가 시 큰 이득).

## 6. 진행상황 및 로드맵

**현재 위치**
- 환경: TVM(표준 API, 소스 빌드) + mysim 빌드 자동화 완료.
- **B0(논리적 동작)**: 단일 Llama 레이어를 끝까지 컴파일·실행, 상대오차 0.13% — **완성**.
- **B1(하드웨어 타일링)**: 임의 차원을 64×64로 쪼개 누적 — **완성**(M·N·K 모두).

**다음 단계**
1. **정적 비용 분석 도구**: 명령어 수·메모리·시뮬 출력량을 세어 "실제 3B 한 레이어가 돌아갈 규모인가"를 추정. (성능/시간 예측이 아니라 자원/실행가능성 분석 — 하드웨어 타이밍 정보가 없어 시간 예측은 불가.)
2. **실제 모델 수입(BYOC 방향)**: TVM 프런트엔드로 ONNX/HF 모델을 받아, 지원 부분은 우리 백엔드로·나머지는 CPU로 — §1.5의 목표 그림.
3. (이후) decode 단계, ISA 확장 검토(전치·reduce-max·루프·누산).

**보류**
- mysim 수정이 필요한 ISA 확장(루프/누산/reduce-max)은 현재 보류. 필요성은 위 비용 분석으로 판단.

## 7. 결론

- 소수의 기본 연산(행렬곱·원소연산·제곱근·exp)과 **우회 기법(합·복제·전치·SiLU·RoPE)** 만으로 **전체 Llama 레이어를 NPU 명령어로 생성·실행**하는 데 성공했고(상대오차 0.13%), **임의 차원 64×64 타일링**까지 동작한다.
- 현재 방식(operator-level "lite")은 빠른 검증에 적합했고, **버려지지 않은 채 BYOC의 백엔드로 확장**될 수 있다.
- 앞으로의 방향은 명확하다: **모델 수입은 TVM/BYOC에 기대고, NPU 명령 생성 백엔드는 우리가 유지**한다. tensorize는 이 프로젝트를 정식 TVM 백엔드로 키울 때의 카드다.
