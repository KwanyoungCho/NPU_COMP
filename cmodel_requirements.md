# 제공 NPU c-model(`a_npu/a.out`) 제한사항 및 요구사항

## 1. 개요
- 대상: 제공받은 NPU 실행기 바이너리 `a_npu/a.out`
- 목적: c-model을 사용한 모델(예: Llama) 매핑 과정에서 확인된 **제한사항**과 **기존 코드 수정 요구사항** 정리
- 범위: ① 실행기 구조적 제약 3건 + 해결 방안, ② 미지원 연산 및 지원 요청

---

## 2. 제한사항 및 해결 방안

### 2.1 고정 용량 한계
- **현상**
  - 기존 미팅에서 데이터 및 프로그램 메모리는 **무한하다고 가정**하기로 결론
  - 그러나 실행기 내부에는 **데이터 버퍼 16KB(= FP16 8,192개)**, **명령어 32K개**가 **고정 크기**로 존재
  - 추가 확인 사항
    - 데이터·명령어뿐 아니라 **PE 연산 버퍼(입력1/입력2/출력 등)도 각 16KB로 고정**
    - 데이터 적재 루프가 **8,192개에서 하드 컷** → 입력 파일이 더 커도 초과분은 무시됨
    - 용량 초과 시 **오류 없이 인접 메모리를 침범**(특히 데이터 버퍼 초과분이 **프로그램 메모리 영역을 덮어써** 실행이 망가짐) → 디버깅 곤란
- **해결 방안 (기존 코드 수정)**
  - 데이터 버퍼·프로그램 메모리 배열을 **입력 파일 크기에 맞춰 동적 할당**(또는 충분히 큰 값으로 확장)
  - 데이터 적재 루프의 **8,192 하드 리밋 제거**
  - 주소 접근 시 **경계 검사 추가** → 초과 시 조용한 오염 대신 **명시적 오류 처리**
  - 참고: PE 연산 버퍼는 물리 PE 배열(64×64) 한 타일에 대응하므로 무한 확장 대상이 아님 → 큰 연산은 **타일링 전제** + 경계 검사로 보호

### 2.2 파일 쓰기 기능 부재
- **현상**
  - NPU 실행 후, 데이터 버퍼 내 **결과를 파일로 쓰는 기능이 없음**
  - 따라서 **단계 간 데이터 연결(체이닝)이 불가** (한 실행 결과를 다음 실행 입력으로 넘길 수 없음)
- **해결 방안 (기존 코드 수정)**
  - 실행 종료 시 데이터 버퍼를 **입력과 동일한 FP16 바이너리 포맷으로 파일 출력**하는 기능 추가
  - 형태: 명령행 옵션(예: 출력 파일 경로 지정) 또는 전용 "store-to-file" 동작
  - 효과: 한 단계의 출력 파일을 다음 단계 입력으로 사용 → **다단계 파이프라인 구성 가능**

### 2.3 프로그램 카운터 무한 증가
- **현상**
  - 이전 버전: 프로그램 카운터가 **40개만 실행 후 종료**
  - 현재 버전: 프로그램 카운터가 **무한 증가** → 출력이 끝없이 발생하여 **결과 확인이 까다로움**
- **해결 방안 (기존 코드 수정)**
  - **명확한 종료 조건 추가**
    - (a) **종료(HALT/END) 명령어 도입** → 해당 명령을 만나면 실행 중단, 또는
    - (b) **로드된 명령어 개수(프로그램 길이)에 도달하면 종료** (32K 전체를 순회·반복하지 않도록)
  - 효과: 출력이 유한해지고 **결과 캡처가 용이**

## 3. 미지원 연산 및 지원 요청
> 모델 매핑에 필요하나 현재 ISA에 없는 연산. 대부분 기존 명령 조합으로 **우회 가능**하나 명령어 수가 증가하며, 일부는 **우회 불가**하여 **네이티브 지원이 필요**함.

### 3.1 우회 구현 가능 (네이티브 지원 시 효율·정확도 개선)
| 필요 연산 | 사용처 | 현재 대체(우회) 방법 |
|-----------|--------|----------------------|
| **Reduce Sum** (행 합산) | RMSNorm(제곱합→평균), Softmax(지수합) | 1로 채운 벡터와 **내적(행렬곱)** |
| **Broadcast** (스칼라→벡터) | RMSNorm(1/rms를 채널에 곱), Softmax(지수합으로 나눔) | 1 벡터와 **행렬곱(외적)** 으로 값 복제 |
| **Transpose** | Attention의 Q×Kᵀ | ① 호스트에서 **미리 전치 배치** ② 런타임은 **원소 단위 복사** |
| **부호 반전(negate)** | RoPE(rotate_half의 −x) | **0 벡터에서 뺌** (0 − x) |
| **메모리 간 복사(move)** | RoPE(rotate_half의 절반 복사) | **0 벡터와 더함** (0 + x) |
| **sin/cos** | RoPE(위치별 회전각) | 호스트에서 **사전 계산해 메모리에 배치** |
| **표준 활성화(SiLU/sigmoid)** | SwiGLU FFN | 현재 활성화가 **x²·sigmoid(x)** 형태로 부적합 → **exp·덧셈·나눗셈·곱셈 조합**으로 SiLU 구성 |

- 위 연산들은 동작은 가능하나 **연산당 명령어 수가 늘어** 효율이 저하됨 → 네이티브 명령 추가 시 명령어 수·정확도 개선

### 3.2 우회 불가 (지원 필수)
| 필요 연산 | 사용처 | 현황 |
|-----------|--------|------|
| **Reduce Max** (행 최댓값) | Softmax 최댓값 빼기(오버플로 방지) | **대체 방법 없음** — 현재는 weight를 작게 설정해 overflow 회피 |

- **상세**: Softmax는 수치 안정화를 위해 각 행의 최댓값을 먼저 빼야 함(`exp(x−max)`). 그러나 행 최댓값을 구하는 reduction이 ISA에 없어 **우회가 불가능**
- **영향**: 점수가 일정 수준(약 11) 이상이면 `exp`가 **FP16 표현 한계(65,504)를 초과 → inf/NaN** → 결과 손상
- **요청**: **Reduce Max(행 방향 최댓값) 명령 지원이 필수** (최우선)

### 3.3 지원 요청 우선순위
| 우선순위 | 연산 | 사유 |
|----------|------|------|
| 1 (필수) | **Reduce Max** | 우회 불가, Softmax 안정성·정확성 직결 |
| 2 | Reduce Sum / Broadcast | 빈번 사용, 명령어 수 대폭 절감 |
| 3 | Transpose | 우회 시 명령어 폭증·레이아웃 제약 해소 |
| 4 | 표준 활성화(SiLU 등) | FFN 간결화·정확도 향상 |
| 5 | negate / move / sin·cos | 편의·소폭 효율 개선 |

---

## 4. 요약
- **구조적 제약 3건**은 모두 **기존 코드 수정으로 해결 가능**
  - 고정 용량 → 동적 할당·확장 + 경계 검사
  - 파일 쓰기 부재 → 결과 파일 출력 기능 추가
  - PC 무한 증가 → 종료 조건(HALT 또는 프로그램 길이) 추가
- **미지원 연산**은 대부분 우회 가능하나, **Reduce Max만은 우회 불가 → 네이티브 지원 필수**
- 그 외 Reduce Sum·Broadcast·Transpose·표준 활성화 등의 네이티브 지원 시 명령어 수·정확도 개선
