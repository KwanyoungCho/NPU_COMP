# NPU c-model 재구현 및 Llama 레이어 매핑 보고서

## 1. 개요
- **목적**: 소스 없는 NPU c-model 바이너리에 Llama3 transformer 레이어를 매핑
- **접근 단계**
  - ① 기존 바이너리 분석 → ② 동일 동작 시뮬레이터 재구현 → ③ 그 위에서 Llama 레이어 코드 생성·검증
- **핵심 성과**
  - 명령어 체계(ISA) 완전 역추출 및 **byte-exact 재구현**(예제 55개 전부 1바이트도 불일치 없음)
  - 원본의 3대 구조적 한계를 비파괴적으로 해소한 개선판 확보
  - Llama3 레이어를 NPU 명령어로 생성·검증(상대오차 0.12%)하여 매핑 실현 가능성 입증
- **본 보고서의 핵심**: ISA가 **지원하지 않는 연산을 어떻게 우회**했는지(§5), 그리고 **꼭 필요하거나 추가하면 좋을 명령어**(§6) 정리

## 2. 기존 바이너리 c-model 현황 및 문제점

### 2.1 현황
- 형태: **소스·문서 없는 컴파일된 실행 바이너리**
- 입력: 명령어 파일, 데이터 파일(FP16)
- 출력: **stdout 텍스트 trace만** (명령어별 PE 입출력 값 나열)
- 실행: 외부 라이브러리(특정 conda 환경) 의존

### 2.2 문제점
- **고정 용량 한계**: 데이터 버퍼 8,192개(FP16), 명령어 32,768개로 하드 고정 → 큰 모델 수용 불가
- **결과 영속화 불가**: 파일 쓰기 기능 부재 → 결과 저장 불가, 단계 간 데이터 연결(체이닝) 불가
- **자체 종료 없음**: 프로그램 카운터 무한 증가 → **출력 무한 폭주**(디스크 고갈 위험), 캡처에 우회 필요
- **안전장치 부재**: 용량 초과 시 오류 대신 **조용한 메모리 오염** → 디버깅 곤란
- **이식성 저하**: 라이브러리 경로 의존, 무한 루프로 출력 캡처가 까다로움
- **종합**: 동작 검증·확장·실모델 적용이 모두 어려운 상태

## 3. 동일 동작 c-model 재구현

### 3.1 ISA 역추출
- 제공된 **명령어 예제 55개 + 명령어 생성기 소스**에서 32비트 명령어 인코딩 규칙 추출
- 명령어군 식별
  - 벡터 연산 13종: 사칙연산, 논리(AND/OR/NOT/XOR/NAND/NOR), 시프트, 비교, min/max, sqrt, exp, 형변환
  - 행렬 연산 4종: 가산·감산·곱셈·이동(+ 활성화 옵션)
  - 제어: 주소 설정, 길이 설정, 타일 설정, 적재(load), 저장(save)

### 3.2 핵심 동작 규명
- **수치 정밀도**: 내부 연산은 **float32**, FP16 반올림은 **데이터 저장 시점에만** 발생
  - 근거: 나눗셈·제곱근·지수 결과가 full 정밀도로 출력, 저장 후 재적재 시 값이 미세 변화
- **활성화 함수**: 표준 함수가 아닌 **x² · sigmoid(x)** 형태임을 수치 대조로 규명
- **표기 규칙**: 명령어 값은 16진수, 데이터 값은 10진수 등 출력 포맷까지 일치 확인

### 3.3 재구현 및 검증
- **C++로 시뮬레이터 재작성**(명령어 해독 + 실행 + 동일 trace 출력)
- 원본 캡처 기준 출력(golden)과 자동 대조 → **예제 55개 전부 byte-exact 일치**
- 가장 우려되던 FP16 수치까지 정확히 일치

### 3.4 한계 해소(개선) — 기존 검증을 깨지 않는 비파괴적 추가
- **용량 확장**: 버퍼·명령어 고정 한계 제거(파일 크기 기반 동적 할당·자동 확장)
- **결과 파일 출력(write-back)**: 실행 후 데이터 메모리를 FP16 파일로 저장 → 단계 간 체이닝 가능
- **정상 종료(halt)**: 종료 명령 추가 → 무한 출력 제거, 캡처 용이
- 외부 라이브러리 의존 제거, 원본과 trace 동일 유지 확인

## 4. Llama 레이어 실행 예제

### 4.1 전제 및 환경
- **하드웨어 사양 반영**: 벡터 연산기 256개, 행렬 PE 4,096개(64×64 배열로 추정)
- **가정**: 우선 **타일링 없이 한 번에 처리(one-shot)**, **축소 차원**(명령어 인코딩 한계 255 이하)
- **도구**: Python 어셈블러/코드 생성기 구축(명령어 인코딩·메모리 정적 배치·시뮬레이터 실행·결과 비교 자동화)

### 4.2 구현 구성요소
- **행렬곱**(선형변환), **RMSNorm**(정규화)
- **Attention**: GQA(그룹 쿼리), RoPE(회전 위치 인코딩), causal softmax
- **SwiGLU FFN**, **Residual**(잔차 연결)
- 흐름: 입력 → RMSNorm → Attention → 잔차 → RMSNorm → FFN → 잔차 → 출력

### 4.3 검증·실행 결과
> 모든 결과는 **독립 구현한 float 참조와 교차 비교**(differential test). byte-exact = 1비트도 불일치 없음, 그 외는 FP16 반올림 수준 오차.

**(a) 시뮬레이터 재구현 검증 — 원본 명령어 예제 55종**

| 항목 | 결과 |
|------|------|
| 예제 55종 trace 일치 | **PASS 55 / FAIL 0 (전부 byte-exact)** |

**(b) Llama 커널별 코드젠 검증** (축소 차원, 1-shot)

| 커널 | 차원 | 불일치 | 최대오차 | 명령어 수 | 판정 |
|------|------|--------|----------|-----------|------|
| 행렬곱(matmul) | 8×64 @ 64×64 | 0 / 512 | 0 | 13 | **byte-exact** |
| RMSNorm | [8, 64] | 182 / 512 | 0.00195 | 673 | FP16 일치 |
| Attention(1-head, causal) | seq8, hd16 | 0 / 128 | 0 | 135 | **byte-exact** |
| SwiGLU FFN | [8, 64]→128 | 1 / 512 | 0.0156 | 97 | FP16 일치 |

**(c) 전체 Llama3 레이어**

| 항목 | 값 |
|------|-----|
| 구성 | SEQ8, D64, H4, KV2, HD16, F128 (RMSNorm→GQA+RoPE+causal→잔차→RMSNorm→SwiGLU→잔차) |
| 명령어 수 | 6,999 |
| 최대 절대오차 | 0.0037 |
| **상대오차** | **0.12%** (FP16 수치 노이즈 수준) |
| 판정 | **PASS** |

**(d) 원본 바이너리 교차검증** — 축소 attention 커널을 **원본 a.out에서도 실행**
- 원본 trace의 최종 출력과 mysim 결과 비교 → **차이 = 1 FP16 ULP(0.00195)**
- 이 차이는 연산 오류가 아니라 **원본 stdout(float32 표시값) vs mysim 저장값(FP16 양자화)** 의 표현 차이 → 같은 값
- 결론: 코드젠 산출물이 원본 시뮬레이터에서도 동일 동작 확인

**(e) 프로그램별 G-buffer 사용량** — 원본(8,192 FP16 한계) 실행 가능 여부

| 프로그램 | G-buffer(FP16) | 원본 a.out 실행 |
|----------|----------------|-----------------|
| 행렬곱 | 5,120 | ✅ 가능 |
| RMSNorm | 1,796 | ✅ 가능 |
| Attention(1-head) | 2,256 | ✅ 가능 |
| SwiGLU FFN | 34,816 | ❌ 8,192 초과 |
| 전체 레이어 | 54,036 | ❌ 8,192 초과 |

→ 작은 커널은 원본에서도 검증되나, FFN·전체 레이어는 G-buffer 한계를 넘어 **개선판(mysim)에서만 실행 가능**.

## 5. [핵심] 미지원 연산의 우회 구현
> ISA에 대응 명령이 없는 연산을, 기존 명령 조합으로 어떻게 구현했는지 정리. (비용 = 추가되는 명령어 수·제약)

| 필요 연산 | 사용처(레이어 연산) | ISA 지원 | 우회 방법 | 비용·한계 |
|-----------|-------------------|---------|-----------|-----------|
| **합 reduction** (행 합산) | RMSNorm(제곱합→평균), softmax(지수합→분모) | ✗ | **1로 채운 벡터와 행렬곱** → 내적이 곧 합 | reduction마다 행렬곱 1회 추가(정규화·softmax에서 빈번) |
| **브로드캐스트** (스칼라→벡터) | RMSNorm(1/rms를 채널에 곱), softmax(행 합으로 나눔) | ✗ | **1 벡터와 행렬곱**으로 값 복제 | 추가 행렬곱 1회 |
| **전치(transpose)** | Attention Q·Kᵀ (K를 [HD,SEQ]로, K projection용 입력 전치) | ✗ (strided 접근도 없음) | ① 가중치를 **호스트에서 미리 전치 배치** → Q·Kᵀ를 런타임 전치 없이 계산 ② 런타임 전치는 **원소 단위 복사** | 데이터 레이아웃 제약 + 원소별 복사 시 명령어 폭증(O(행×열)) |
| **SiLU 활성화** | SwiGLU FFN(게이트 활성화) | ✗ (HW 활성화는 x²·sigmoid로 부적합) | **exp·덧셈·나눗셈·곱셈 조합**으로 z/(1+exp(−z)) 직접 구성 | 활성화 1개에 명령어 5~6개 + 중간 버퍼 다수 |
| **부호 반전(negate)** | RoPE(rotate_half의 −x), SiLU(exp(−z)의 −z) | ✗ | **0으로 채운 벡터에서 빼기**(0−x) | 0 벡터 상수 1개 점유 |
| **메모리 간 복사(move)** | RoPE(rotate_half 절반 복사), 전치, 누산기 0 초기화 | ✗ (load→save 직결 불가) | **0을 더해서**(x+0) 통과 | 사소한 명령어 추가 |
| **softmax 최댓값 차감** | Attention softmax(수치 안정화) | ✗ (행 최댓값 reduction 없음) | **우회 불가** → 현재 생략, 작은 가중치로 회피 | 점수 과대 시 **FP16 지수 오버플로(NaN)** — 정확성 결함 |
| **sin/cos** | RoPE(위치별 회전각) | ✗ | 위치 의존 상수이므로 **호스트에서 사전 계산해 적재** | 고정 위치엔 충분, 온디바이스 동적 생성 불가 |
| **분수 스케일·eps 가산** | RMSNorm(eps), Attention(1/√d 스케일) | ✗ (즉시값이 정수만) | 정수 나눗셈으로 대체(예 ÷√d), eps는 현재 0 | 임의 분수 상수 인코딩 불가, 정밀도 일부 손실 |
| **메모리 동적 할당** | 전 구간(모든 중간 버퍼·활성값) | ✗ | 코드젠이 **모든 버퍼 주소를 정적 사전 배치** | 차원 변경 시 재생성 필요 |

**요약**: 합·브로드캐스트·전치·SiLU·negate·복사 등은 **기존 명령 조합으로 우회 가능**하나 명령어 수가 늘어남. 단 **softmax 최댓값 차감만은 우회 수단이 없어 현재 미해결**(정확성 측면 최우선 보완 대상).

## 6. [핵심] 필요·추가 권장 명령어
> 우선순위별로 정리. (현재 우회 방식 → 추가 시 효과)

### 6.1 필수 — 정확성/안정성에 직결, 우회 불가
- **행 최댓값 reduction (row-max)**
  - 현재: 우회 수단 없음 → softmax 최댓값 차감 생략 → 점수가 커지면 FP16 지수 오버플로(NaN)
  - 추가 시: **softmax 수치 안정화** → 실제 가중치 크기에서도 안전, 현재 유일한 정확성 결함 해소
  - **권장 1순위**

### 6.2 고가치 — 명령어 수·정확도 대폭 개선(우회는 가능)
- **행 합 reduction (row-sum)**
  - 현재: 1 벡터와 행렬곱으로 우회(정규화·softmax에서 매번)
  - 추가 시: 행렬곱 호출 제거 → 명령어 절감, PE array 점유 감소
- **전치 명령 또는 전치/strided load**
  - 현재: 데이터 사전 전치 + 원소별 복사(명령어 폭증, 레이아웃 제약)
  - 추가 시: attention Q·Kᵀ 자연 처리, **명령어 대폭 절감**, 데이터 배치 자유도 확보
- **표준 활성화 명령 (SiLU / sigmoid / GELU 등)**
  - 현재: HW 활성화(x²·sigmoid)가 부적합 → 다수 명령 조합으로 SiLU 구성
  - 추가 시: FFN 간결화, 정확도 향상, 다양한 모델 지원
- **reduction의 일반화(축·연산 선택형: sum/max/mean)**
  - 위 row-sum·row-max를 하나의 reduction 명령군으로 제공하면 정규화·softmax·attention 전반이 단순화

### 6.3 편의·효율 — 소폭 개선
- **reciprocal / rsqrt(1/√x)**: RMSNorm의 sqrt+나눗셈을 단일 명령으로(표준 정규화 패턴 직접 지원)
- **부호 반전(negate) / 메모리 복사(move)**: 0 빼기·0 더하기 트릭 제거
- **브로드캐스트 명령**: 1 벡터 행렬곱 대체
- **부동소수점/광폭 즉시값**: eps(1e-5), 1/√d 등 분수 상수 직접 인코딩 → 정수 트릭 제거, 정밀도 확보
- **sin/cos 명령**: 온디바이스 RoPE 생성(동적 위치 대응)

### 6.4 ISA 구조 차원(별도 검토)
- **루프/분기 지원**: 현재 완전 언롤 → 프로그램 거대화·차원별 재생성. 도입 시 프로그램 크기·재사용성 대폭 개선
- **광폭 타일 차원 인코딩 또는 타일링 가속**: 현재 타일 차원 8비트(≤255). 실제 차원은 소프트웨어 타일링으로 해결 가능하나, 인코딩 확장/자동 타일링이 있으면 코드젠 단순화

### 6.5 우선순위 요약
| 순위 | 명령어 | 효과 | 우회 가능? |
|------|--------|------|-----------|
| 1 | **행 최댓값 reduction** | softmax 안정화(정확성) | ✗ (불가) |
| 2 | 행 합 reduction | 명령어 절감 | △ (행렬곱 우회) |
| 3 | 전치 / 전치 load | 명령어 폭증 해소 | △ (복사 우회) |
| 4 | 표준 활성화(SiLU 등) | FFN 간결·정확 | △ (조합 우회) |
| 5 | rsqrt·negate·move·FP 즉시값 | 효율·편의 | △ (트릭 우회) |
| 별도 | 루프/분기, 광폭 타일 인코딩 | 구조적 개선 | ― |

## 7. 현재 제약 및 향후 과제
- **softmax 안정성**: 행 최댓값 reduction 부재 → 현재 작은 가중치로 회피. **row-max 명령 추가가 근본 해결책**(§6.1)
- **전체 차원 미지원(타일링)**: 타일 차원 8비트(≤255)·버퍼 용량 제약 → 실제 Llama3 차원은 **64×64 타일링 + 256-lane 벡터 타일링** 구현 필요(현재 의도적 보류)
- **정적 특화**: 루프 부재로 차원(특히 시퀀스 길이)별 프로그램 재생성 필요 → **최대 길이 1회 생성 + 마스킹**으로 실용적 해결 가능
- **명령어 효율**: 우회 구현으로 인한 명령어 증가 → §6 명령어 추가 시 대폭 개선

## 8. 결론
- 소스 없는 바이너리에서 **ISA 완전 역추출 및 byte-exact 재구현** 달성(55/55)
- 원본의 용량·결과저장·종료 한계를 **비파괴적으로 해소**
- 그 위에서 **Llama3 레이어를 코드 생성·검증**(상대오차 0.12%)하여 NPU 매핑 경로 입증
- **핵심 시사점**
  - 현 ISA로도 대부분 연산을 **우회 구현 가능**하나 명령어 수가 증가
  - **단 softmax 최댓값 차감은 우회 불가** → **행 최댓값 reduction 명령 추가가 최우선**
  - 전치·표준 활성화·합 reduction 명령 추가 시 **명령어 수·정확도 동시 개선**
  - 향후 타일링·reduction 명령 보강으로 실제 모델 전체 차원 확장 가능
